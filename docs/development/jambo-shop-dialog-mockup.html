Label shows raw numbers: 9.6 + (+0.2) kg · max 15.0 kg. Bar color switches: <75% → lime; 75–90% → orange; >90% → red. Pending segment always orange regardless of fill color.
Buyback section: Appears above regular stock when player has recently sold items. Cyan accent — ↩ BUYBACK badge on each row. Price is the sell price Jambo paid (not the original item value). Section subheader shows "until next beat" caveat. Flushed automatically when the game beat advances.

Backend Research Note — Buyback System
⚠ Research required before planning implementation
What needs to exist backend-side:
1. A transient buyback ledger per merchant session — list of { item_id, item_data, qty, buyback_price, beat_acquired } records.
2. The ledger is populated when ShopInterface.sell() is called — item transferred to Jambo's possession at a tagged price equal to what the player received.
3. On every game_tick_events() call, stale ledger entries (where beat_acquired < current_beat) are flushed and the items enter regular inventory rotation.
4. The merchant serializer must expose buyback items as a separate list (e.g. buyback_items: [...]) distinct from stock, so the frontend can render them with the buyback treatment.
5. New API endpoint needed: POST /api/shop/buyback — takes { merchant_id, item_id }, validates the ledger entry, transfers item back to player at buyback price.

Files to investigate: src/npc/_shop.py (MerchantShopMixin), src/interface.py (ShopInterface), src/api/routes/ (no shop routes exist yet), src/api/services/game_service.py.
