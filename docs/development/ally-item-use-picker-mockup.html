Combined card logic
Each effect in item.effects[] array renders independently — order: heal bars first, then chips (removal → application → attr)
Removal chips are conditionally styled: chip-remove if the target has the status, chip-inert (strikethrough) if not — no surprises
Application chips check member.states: if already active → chip-refresh; otherwise → chip-apply
New item types never require a new card layout — just add a new entry to the effects array; the chip row handles it
Implementation Notes (Updated)
Backend — composable effects array: Replace the single effect field with an effects: [] array in InventoryItemSerializer.serialize(). Each element has a type discriminator:
{ "type": "heal", "stat": "hp", "power": 25, "range": [20, 30] }
{ "type": "heal", "stat": "fatigue", "power": 100, "range": [80, 120] }
{ "type": "status_remove", "status_name": "Poisoned", "status_type": "poison" }
{ "type": "status_apply", "status_name": "Hawkeye", "duration": 30 }
{ "type": "attr_buff", "stat": "strength", "label": "STR", "amount": 5, "duration": 25 }
Frontend rendering loop: Iterate item.effects. For each: "heal" → render a stat bar row with projected gold segment; "status_remove" → render red chip (check target's member.states for styling); "status_apply" → render green chip (check if already active → purple refresh chip); "attr_buff" → render orange/cyan chip (per-stat color table).
Extensibility: New item types in Python only require a new type string and a corresponding chip renderer on the frontend — no new card layout, no prop threading.
Fallback: If item.effects is absent or empty, render current HP bar only — no regression for items without serialized effect data.
State data source: The fresh party fetch (/player/status) already runs when "Use on..." is clicked — extend the party member data to include states: ["Poisoned", ...] so the frontend can conditionally style chips per-target.
